## OTF eval-key `a` keygen RTL Walkthrough

이 문서는 `MOAI_GPU/verilog/`에 추가한 샘플 RTL을 **모듈별로 순서대로** 설명합니다.\n+“PRNG가 왜 이렇게 구성되는지(SHAKE128 → reject sampling → chunk streaming)” 원리와,\n+각 모듈의 **핸드셰이크(valid/ready), 상태기계, backpressure가 어디서 생기는지**를 연결해서 읽을 수 있게 하는 것이 목표입니다.\n+\n+> 주의: 이 RTL은 “설계 스케치/교육용”입니다. Keccak/나눗셈/모듈러는 비최적이며,\n+> 실제 하드웨어에서는 Barrett/Montgomery, 파이프라이닝, 멀티-lane 스케줄링 등이 필요합니다.\n+\n+---\n+\n+## 0) 큰 그림: 왜 `a`를 on-the-fly로 만들 수 있나?\n+\n+eval-key(relin/galois key)는 RLWE 샘플 (b,a)들의 묶음이며, 여기서\n+\n+- `a`는 각 limb modulus \(q_j\)에 대해 **균일 랜덤 다항식**\n+- `b`는 대략 `-(a*s + e)`에 타깃항(예: \(s^2\) digit)이 더해진 값\n+\n+입니다. 따라서 키 저장 관점에서 보면 **`a`는 seed/메타데이터로부터 재생성 가능한 랜덤 값**이라\n+메모리에서 매번 읽지 않고 생성(OTF)하여 downstream(키스위칭의 evk mult)에 공급할 수 있습니다.\n+\n+이 RTL은 그 OTF `a` 생성의 하드웨어 구성을 보여줍니다.\n+\n+---\n+\n+## 1) PRNG 원리: SHAKE128 XOF → 64-bit word 스트림\n+\n+### SHAKE128이 하는 일\n+SHAKE128은 Keccak-f[1600] permutation을 기반으로 하는 XOF(extendable output function)입니다.\n+\n+- 입력(seed/descriptor)을 absorb(흡수)하여 내부 1600-bit state를 만든 뒤\n+- 원하는 만큼의 출력을 squeeze(짜냄)\n+\n+합니다.\n+\n+SHAKE128의 **rate**는 168 bytes(1344 bits)라서, 한 번의 permutation 결과에서\n+168B를 출력으로 사용하고, 더 필요하면 permutation을 다시 돌려 다음 블록을 얻습니다.\n+\n+이 RTL에서는 squeeze 결과를 64-bit word로 쪼개서 **연속 스트림**으로 downstream에 넘깁니다.\n+\n+---\n+\n+## 2) Uniform modulo 샘플링: reject sampling\n+\n+### 왜 reject sampling이 필요한가?\n+`x % q`로 단순히 줄이면, `x`가 균일한 64-bit여도 \(q\)가 2의 거듭제곱이 아니면\n+나머지 분포에 **bias**가 생깁니다.\n+\n+이를 막기 위해, 아래 조건을 만족하는 `x`만 accept 합니다:\n+\n+\\[\n+T = \\left\\lfloor \\frac{2^{64}}{q} \\right\\rfloor \\cdot q\n+\\]\n+\n+- `x < T` 이면 accept, 출력 `x % q`\n+- 아니면 reject 후 다음 `x`로 반복\n+\n+이렇게 하면 `0..q-1`의 분포가 균일해집니다.\n+\n+**하드웨어 관점**에서 중요한 점:\n+- reject가 발생하면 “추가 word가 더 필요” → SHAKE/bitFIFO/샘플러에 부하 증가\n+- 따라서 throughput은 `accept_rate`에 의해 감소 가능\n+\n+---\n+\n+## 3) Chunk streaming이 왜 필요한가?\n+\n+키스위칭(evk mult)은 `a`를 폴리 전체가 다 완성될 때까지 기다릴 필요 없이,\n+**일부 chunk만 있어도 바로 곱/누적을 진행**할 수 있습니다.\n+\n+그래서 생성기는 `CHUNK_SIZE`개 coeff를 모아 AXI-Stream으로 내보내고,\n+consumer의 `tready`에 의해 backpressure가 전달됩니다.\n+\n+---\n+\n+## 4) 모듈별 설명\n+\n+### 4.1 `otf_keygen_pkg.sv`\n+**역할**: 전체 설계의 파라미터/타입 정의.\n+\n+- `COEFF_W` (기본 64): coeff를 u64로 전달 (실제론 q 비트폭만 필요)\n+- `CHUNK_SIZE` (기본 32): AXI-Stream 1 beat에 포함되는 coeff 개수\n+- `prng_req_t`: request descriptor\n+  -
